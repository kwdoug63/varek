# VAREK v1.3 — Design Notes

**Status:** Exploratory roadmap. Design directions for v1.3 derived from v1.2 RFC §7 (Open Questions). Not committed; subject to revision as v1.2 reaches GA and operator evidence accumulates. This document establishes the architectural intent under which v1.3 design work proceeds; full RFCs follow per item.

**Author:** Kenneth Wayne Douglas, MD
**Last updated:** April 28, 2026

---

## 0. Scope of this document

v1.2 explicitly defers seven items to v1.3 or later. This file sketches the architectural direction VAREK intends to take on each, in advance of dedicated RFCs. The sketches are deliberately short and option-shaped: they fix the problem statement and the candidate approaches, not the final design. Each item will be promoted to its own RFC before implementation, on the same review pattern v1.1 and v1.2 used.

Three constraints carry forward from v1.2 and apply to all v1.3 work:

- v1.2's data model and evaluator extend forward without breaking changes. Schema enums (decision types, scopes, match predicates) are open; v1.3 additions are non-breaking.
- The deployment model in v1.2 §1.1 is fixed: standalone runtime, thin framework adapters, no policy enforcement outside VAREK's process space. v1.3 does not relax this.
- Fail-closed semantics extend to every new code path. Any v1.3 mechanism that introduces a new failure mode must specify a fail-closed default before it ships.

## 1. Semantic action derivation

**v1.2 position (§2.1):** transitions are syscall-layer only. Semantic actions (`HTTPRequest`, `WriteFile`, `LLMInvocation`) are not part of the v1.2 vocabulary.

**v1.3 problem statement.** Operators want to author policies in domain language (`HTTPRequest(url=...)`) rather than syscall language (`connect(af=AF_INET, addr=...)`). The translation problem is non-trivial: a TLS-encrypted `write()` on a socket fd does not, by itself, surface the request URL; a sequence of `openat`, `read`, `close` does not, by itself, indicate a logical "load model weights" action. Inferring semantic actions from raw syscall streams in the general case is an unsolved derivation problem.

**Candidate approaches.**

- *Explicit emission.* A VAREK-aware client library (or language-runtime hook) emits semantic events to the evaluator alongside the underlying syscall stream. Policies match on the semantic event; syscall-layer rules continue to operate as a backstop. Honest about the mechanism: this is opt-in instrumentation, not inference, and only covers code that uses the library.
- *Pattern-derived inference.* A small set of well-defined syscall-sequence patterns (e.g., `socket → connect → write → read` to a specific port) is canonicalized into named actions. Limited expressiveness, but works without instrumentation. Suitable for a fixed catalog of common actions, not general semantic recovery.
- *Hybrid, defaulting to explicit.* Bundles ship pattern recognizers for a small fixed catalog; instrumented runtimes emit explicit events for everything else. Policy authors see one vocabulary; the evaluator dispatches to the appropriate source.

**Likely v1.3 direction.** Hybrid. Explicit emission as the supported path, pattern inference for a narrow well-tested catalog. The schema gains a `semantic_event` match predicate that operates on either source. Coverage limits are documented; what semantic policies *cannot* see is enumerated, not hidden.

**RFC trigger:** the dedicated RFC opens once v1.2 GA ships and the catalog of operator-requested semantic actions is concrete enough to scope.

## 2. Decision types: Defer and Transform

**v1.2 position (§2.3):** decision types are Allow / Deny / RateLimit only. Defer (human-in-the-loop) and Transform (in-flight modification) are deferred.

**v1.3 problem statement, Defer.** Defer requires an approval protocol the v1.2 evaluator does not have. Open questions: who is the approver (an identity, a group, an external service)? what is the approval surface (queue, webhook, in-deployment UI)? what is the timeout policy (fail-deny, fail-allow, escalate)? what happens to the suspended caller during the wait (blocked thread, async return, partial result)? what is the audit trail of an approved Defer decision?

**v1.3 problem statement, Transform.** Transform requires modifying syscall argument data while the calling thread is suspended. Technically feasible via `SECCOMP_NOTIF_IOCTL_ADDFD` (for fd substitution) and `process_vm_writev` (for argument-buffer modification), but the ergonomics are research-grade: any transform must preserve syscall ABI, must not introduce TOCTOU windows, and must be auditable. The set of safely transformable syscalls is small.

**Candidate approaches, Defer.**

- *Webhook-based.* Suspended transition emits a POST to an operator-configured endpoint with the transition record; endpoint returns Allow / Deny within a timeout. Simplest model. Operator carries responsibility for the approval surface.
- *Built-in approval queue.* VAREK ships a minimal approval service (CLI + REST) that holds pending Defers and accepts approver decisions. More opinionated but reduces operator burden.
- *Pluggable approver interface.* Operators implement an `Approver` trait/interface; VAREK ships reference implementations (webhook, CLI queue, no-op-deny).

**Likely v1.3 direction, Defer.** Pluggable approver interface with two reference implementations (webhook, CLI queue). Default timeout policy is fail-deny. Approved decisions log the approver identity and rationale alongside the standard decision record. Approval is per-transition, not per-policy; the same transition pattern in two contexts can have different approvers.

**Likely v1.3 direction, Transform.** Restricted scope. v1.3 ships Transform for a small allowlisted set of syscall+argument combinations where the transform is deterministic and ABI-preserving (e.g., redacting fields from a `write()` to an audit-log fd, substituting a sanitized environment buffer in `execve`). General Transform is deferred again. Honest about the limits: Transform that requires understanding application-level protocol semantics is out of scope.

**RFC trigger:** Defer first, in its own RFC. Transform follows as a separate RFC once Defer is in production.

## 3. Stateful predicates

**v1.2 position (§3.3):** the only state v1.2 maintains is rate-limit counters. Policies cannot express "allow X only if Y happened earlier in this execution."

**v1.3 problem statement.** Several common policy patterns require state: "allow `write()` to `/work/report.csv` only if a corresponding `open()` happened in the same execution"; "deny network egress for any execution that has previously read from `/secrets/`"; "limit total bytes written across all `write()` calls within a session." Without state, these are inexpressible.

**Candidate approaches.**

- *Execution-scoped key-value state.* Each execution gets a small in-memory state map. Rules can read and write keys; matches can predicate on key values. Lifetime tied to the execution.
- *Predicate composition over a fixed history window.* Rules can match against the last N transitions in the same execution without exposing a general state surface. Smaller blast radius, less expressive.
- *Declarative state-machine policies.* Each policy declares its states and transitions explicitly; the evaluator tracks state-machine position per execution. Strong typing, formally analyzable, more authoring overhead.

**Likely v1.3 direction.** Execution-scoped key-value state with a bounded size cap (per-execution memory budget enforced by the evaluator) and a documented set of predicate operators (read, write, increment, compare). State-machine policies considered as a v1.3.x extension if authoring patterns warrant it. Cross-execution state is out of scope.

**Performance constraint.** State access stays inside the evaluator; no syscalls for storage, no network. Sub-millisecond evaluation budget from v1.2 §4.4 still applies.

**RFC trigger:** opens after Defer ships, since Defer's audit-trail design informs how state mutations are recorded.

## 4. Policy language vs. YAML

**v1.2 position (§6.2):** authoring is YAML. Higher-level policy languages are deferred pending evidence of authoring friction.

**v1.3 default position.** YAML continues. The bar for introducing a higher-level language is concrete operator evidence — actual authoring sessions where the YAML is the bottleneck — not aesthetic preference. v1.2 ships with a schema validator and a test harness; v1.3 will add dry-run-against-real-traffic mode if not delivered in v1.2.x. The hypothesis being tested: good tooling against YAML is sufficient for the operator population VAREK serves.

**If evidence forces a language.** Candidates, in rough order of preference:

- *Cedar* (AWS / open-source). Mature, type-checked, designed for authorization decisions, has a verifier. Closest semantic fit; adopting it preserves the option of upstreaming policies and benefits from external tooling.
- *Rego* (Open Policy Agent). Larger ecosystem but more general-purpose; the gap between Rego's runtime model and VAREK's evaluator is wider than Cedar's.
- *VAREK-specific DSL.* Last resort. Building a language is expensive and introduces a long maintenance tail. Justified only if no existing language fits and the friction is severe.

**RFC trigger:** only if v1.2 customer evidence shows authoring friction the tooling cannot resolve. Default outcome is no language change.

## 5. Multi-tenant policy isolation

**v1.2 position (§7 open question 5):** tenant separation achievable via deployment-level isolation; not ergonomic.

**v1.3 problem statement.** A platform team wants to run one VAREK deployment serving multiple tenants (engineering teams, customer organizations, regulatory boundaries) where each tenant authors and owns their own policies, and no tenant's policies can affect another tenant's executions.

**Candidate approaches.**

- *Tenant labels as scope.* Each policy declares a `tenant` field in `scope`; evaluator only considers policies whose tenant matches the calling execution's tenant. Simple, fits the v1.2 schema. Vulnerable to misconfiguration: a missing tenant field is a cross-tenant leak.
- *Tenant-scoped evaluator instances.* The evaluator process holds per-tenant policy sets that are fully isolated from each other; no shared rule space. Stronger isolation, higher resource cost (more state per execution).
- *Hierarchical tenants with platform-policies + tenant-policies.* The platform tenant authors policies that apply across all tenants; individual tenants author policies that apply only within themselves. Composition uses `priority_weighted` with platform priority pinned high.

**Likely v1.3 direction.** Hierarchical tenants. Schema gains a required `tenant` field with a reserved `_platform` value; evaluator enforces composition rules at policy load time. Tenant misconfigurations fail policy validation rather than fail at evaluation.

**RFC trigger:** opens when at least one operator has multi-tenant deployment intent concrete enough to validate the design against.

## 6. Policy versioning and migration

**v1.2 position (§7 open question 6):** policies and bundles each carry their own semver; bundle updates are explicit opt-in. The operational flow is not fully designed.

**v1.3 problem statement.** When a Tier A bundle updates from 1.0.0 to 1.1.0 (or 2.0.0), what does the operator do? When a customer-authored Tier B policy updates and is deployed alongside an unchanged bundle, how is the composed behavior verified? When a breaking change is necessary, what does the migration path look like?

**Direction.**

- Bundle updates are *always* explicit opt-in. There is no auto-update path. An operator's deployed configuration pins a specific bundle version.
- Bundle releases include a *changelog with policy-level diffs* (rules added, rules modified, rules removed, decision changes) generated mechanically from the source diff, not hand-written prose.
- For breaking changes (rule semantics that flip a previously Allow case to Deny, or vice versa), the bundle ships a *migration validator* that the operator runs against their current dry-run logs to surface every transition whose decision would change. Operators see the impact before they pin to the new version.
- Customer-authored policies have no migration tooling burden imposed by VAREK; operators own their own change discipline. VAREK's diff and validator tools are available for customers who want them.

**RFC trigger:** before the second Tier A bundle release, since the migration story needs to be in place before any operator has reason to migrate.

## 7. Audit and compliance reporting

**v1.2 position (§7 open question 7):** decision logs exist; full audit-trail and compliance-report story is v1.3.

**v1.3 problem statement.** Operators in regulated environments (HIPAA, PCI-DSS, FedRAMP, defense) need to demonstrate to auditors which policies were in effect during which intervals, with cryptographic evidence the logs were not tampered with. v1.2's structured decision logs are the raw material; v1.3 produces the auditable artifacts.

**Direction.**

- *Append-only signed log chain.* Decision log entries are written to a hash-chained log; each entry contains the hash of its predecessor. The chain head is signed periodically with a deployment key. Tampering is detectable; the chain is portable across log-storage backends.
- *Time-windowed compliance reports.* A reporting tool consumes the signed log chain plus the policy version history and produces a report of the form: "during interval [t0, t1], the following policies were in effect; the following decisions occurred; the following transitions were denied for reason X." Report format is operator-configurable.
- *Compliance-framework mappings.* For specific frameworks (HIPAA technical safeguards, PCI-DSS req. 7, NIST 800-53 AC-3), VAREK ships a reference mapping between policy decisions and the relevant control. Maintained by the VAREK team; explicit about which controls are *in scope* for a VAREK-based attestation and which are not. Out-of-scope controls are named, not hidden.
- *No claim of compliance certification.* VAREK provides evidence; operators and their auditors make the compliance determination. v1.3 does not assert that running VAREK makes a deployment HIPAA-compliant or otherwise.

**Layer discipline.** This work stays inside VAREK's process space. Audit log shipping, retention, and search are operator infrastructure; VAREK exposes the chain and the reporting tool, not a hosted audit service.

**RFC trigger:** opens after at least one operator with concrete compliance pressure is engaged, so the framework-mapping work is grounded in a real attestation requirement rather than speculative coverage.

## 8. Items explicitly out of scope for v1.3

The following remain deferred past v1.3 unless evidence forces reconsideration. Listed here so the v1.3 scope is bounded by what it *isn't* as well as what it is.

- Cross-execution stateful policies (state shared between executions, durable beyond a single sandbox lifetime).
- Hosted policy-distribution service (signed bundle delivery, key management, central registry).
- Content inspection (DPI, payload regex, TLS interception). Same posture as v1.2 §3.3.
- Covert-channel countermeasures.
- Compliance certification of any kind. VAREK provides evidence; certification is operator and auditor work.
- Framework-plugin packaging. Standalone-runtime positioning from v1.2 §1.1 holds.

## 9. Sequencing intent

The seven v1.3 items are not equal in dependency or operator demand. Likely RFC order, subject to evidence:

1. Defer (§2). Highest concrete demand; design is well-understood; unblocks Transform and informs state-machine design.
2. Audit and compliance reporting (§7). Customer-pull from regulated operators; independent of other v1.3 work.
3. Stateful predicates (§3). Builds on Defer's audit conventions.
4. Multi-tenant isolation (§5). Triggered by concrete operator intent.
5. Policy versioning and migration (§6). Triggered by the second Tier A bundle release.
6. Semantic action derivation (§1). Largest open problem; benefits from waiting for v1.2 operator evidence on which actions matter.
7. Policy language (§4). Default outcome is no change; promoted only on evidence.

Transform follows Defer as a separate RFC after Defer reaches production.

## 10. Out-of-scope for this document

This is a design-direction sketch. It is not an RFC, not a commitment to schedule, and not a specification any of the listed items will be implemented from. Each item gets a dedicated RFC before implementation; this document is the index those RFCs will branch from.
