AI pipelines are chaos.
I built a language to fix that.

April 2026  ·  8 min read  ·  Language Design  ·  AI/ML Engineering

scroll

Every production ML pipeline I built required four separate tools that were never designed to talk to each other. That's not a workflow problem. It's an architectural failure — and it's industry-wide.

The problem nobody talks about

Ask any ML engineer what their pipeline looks like in production and you'll get some version of the same answer: Python for the logic, YAML for configuration, JSON Schema for data validation, and Bash to stitch the stages together. Four languages. Four independent failure surfaces. Zero type checking at the boundaries between them.

I built clinical AI pipelines for years. Imaging analysis, EHR processing, multi-modal inference systems. Every one of them was some variation of this same fragmented stack. And every one of them failed in the same ways.

A Python dict that didn't match the schema expected by the next stage — caught at runtime, 45 minutes into a compute job. A YAML config value parsed as a string when the pipeline expected an integer — no error until the model call failed. A schema change in step three that silently broke step seven because nothing verified compatibility at definition time.

These aren't edge cases. They're the operating conditions of production AI.

In medicine, we don't tolerate ambiguity at system boundaries. A medication order has a drug, a dose, a route, and a frequency — all typed, all validated before anything executes.

AI pipelines have no such structure. The cost of that is compounding as these systems move into hospitals, financial institutions, and critical infrastructure. I decided to do something about it.

What I built

VAREK is an open source AI Pipeline Programming Language. Statically typed, compiled to native machine code via LLVM, with pipeline declarations as a first-class language feature. It replaces the entire Python + YAML + JSON Schema + Bash stack with one language where pipelines are not an afterthought — they're the primary design target.

Here's what the same image classification pipeline looks like in the current fragmented stack versus VAREK:

❌ Current stack — 4 files
# input_schema.json (JSON Schema)
# config.yaml (YAML)
# pipeline.py (Python)
# run.sh (Bash)

→ 4 languages
→ Type info lost at every boundary
→ Runtime errors only
→ No step compatibility checks
✓ VAREK — 1 file
schema ImageInput {
  path:  str,
  label: str?,
  width: int
}

pipeline classify {
  source: ImageInput[]
  steps: [
    preprocess
    -> normalize
    -> infer
    -> postprocess
  ]
  output: ClassResult[]
}

The VAREK type checker verifies at compile time that the output type of preprocess matches the input type of normalize, and so on through the entire pipeline. You cannot write a type mismatch that reaches production. The compiler won't allow it.

You don't have to rebuild anything

The first question every engineer asks: what about my existing Python stack? The answer is: it works. All of it.

VAREK — Python interop
import python::numpy as np
import python::torch
import python::transformers::AutoModel
import python::pandas as pd

Any Python library imports natively. Your existing models, datasets, and utilities work without modification. VAREK replaces the glue between the parts — not the parts themselves. The adoption path is incremental: write one pipeline in VAREK, see the type safety working, expand from there. You are never required to abandon your existing stack.

Same answer for YAML, JSON Schema, and Bash. VAREK reads YAML files directly through var::io. Schema declarations replace Pydantic and JSON Schema with compile-time enforcement. Pipeline orchestration replaces Bash with typed, auditable step sequencing. Migrate one piece at a time. No deadline.

What shipped

659 Tests passing
261 Stdlib functions
10–40× Faster than Python

VAREK v1.0 ships with a full standard library across seven modules: var::io, var::tensor (NumPy-backed), var::http, var::async, var::pipeline, var::model, and var::data. A package manager with 20 commands. An interactive REPL. A source formatter. A documentation generator. And a formal RFC governance process for community language proposals.

The 10–40x speed figure comes from native LLVM compilation benchmarks against equivalent CPython implementations on compute-heavy pipeline workloads. Interpreted mode — which runs without LLVM compilation — is comparable to Python and available for prototyping and development. Production pipelines run compiled.

Why a language and not a library

This is the question I get most often, and it's the right one to ask.

Libraries add APIs. Languages change what's expressible. You cannot add compile-time pipeline type checking as a Python library — it requires a type system that understands pipeline semantics at the grammar level. You cannot enforce that the output type of step A matches the input type of step B in a Python decorator. You can only check it at runtime, after the pipeline has already started running.

VAREK catches these errors when you write the code. Not when you run it. Not when you deploy it. Not 45 minutes into a compute job. When you write it. That requires a language, not a library.

Python remains part of the stack. VAREK replaces the glue, not the ecosystem.

The medical origin

I'm a physician. I trained at USC Keck School of Medicine. I understand, at a clinical level, what happens when a system fails silently at a critical boundary. I've seen it in medical software. I've seen it in every production AI pipeline I've worked on since.

The design philosophy of VAREK came directly from medicine. In a hospital, a medication order has a drug, a dose, a route, and a frequency. The structure enforces correctness at the point of entry. You cannot submit a medication order with a missing field or a wrong type. The system won't allow it.

AI pipelines have no such enforcement. The stakes are rising fast as these systems move into clinical, financial, and infrastructure settings. VAREK is my attempt to give AI pipelines the same structural guarantees that medical systems take for granted.

What I need from you

VAREK is open source under the MIT license. The repo is live, the releases are tagged, the spec paper is published. The language is real and it works.

What I need is engineers who build AI pipelines in production — people who live with the Python + YAML + JSON + Bash fragmentation every day — to try it, break it, and tell me what's missing. File an issue. Open an RFC. Propose a change to the standard library. That's how languages improve.

The first RFC from the community gets a personal response from me within the same day. I mean that.
